# PR-181: JRA unified route, display, and acquisition plan

## Current correction

The previous path was incomplete because it treated acquisition verification as if it could be separated from calendar display and automatic next-fetch construction.

This PR fixes that planning mistake.

## Current PR

This PR defines a corrected JRA completion plan.

The plan checks A first, then downgrades to B+ or B from the same evidence if A is not possible.

It also makes calendar display, day-page display, automatic acquisition, parser integration, and CI validation mandatory before JRA can be considered complete.

## Required completion gates

- acquisition verification
- normalization
- calendar display
- day-page display
- automatic next-fetch route
- CI validation
- rank gate

## Correct rank flow

1. Check A first.
2. If A fails, decide B+ from the same evidence.
3. If B+ fails, decide B from the same evidence.
4. If B fails, keep C or move to NAR.

## Display requirement

A, B+, or B cannot be considered complete unless the data appears in:

- monthly calendar display
- day page display

Required visible fields include rank, source URL, status, first/last race time where applicable, and race rows where available.

## Automatic acquisition requirement

A, B+, or B cannot be considered complete without a repeatable route:

```text
date -> JRA calendar URL -> fetch official page -> parse sections -> split meetings -> normalize rows -> write generated JSON -> render calendar/day pages -> validate in CI
```

## Safety

- Do not mark JRA complete at acquisition-result stage only.
- Do not run a separate late A-rank investigation after finishing B/B+.
- Do not promote records that are not shown in calendar and day-page UI.
- Do not promote records without a repeatable next-fetch route.
- Do not promote records without CI guarding section splits, source URLs, and rank requirements.
