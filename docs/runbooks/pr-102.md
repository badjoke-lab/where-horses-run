# PR-102 — Major-country timetable ingestion v0 plan

## 1. What PR-102 does

PR-102 starts production timetable ingestion planning for 13 countries and all active acquisition groups. It does not complete the timetable UI.

PR-102 locks the static/manual-driven ingestion model for the 13-country timetable MVP. The plan uses `data/static/major-country-acquisition-source-groups.json` as the source of truth for countries, systems/operators, acquisition groups, and official source families.

The locked target scope is:

- 13 countries.
- 24 active acquisition groups.
- Singapore as legacy/no active racing.
- Static/manual-driven timetable v0 first.
- Parser, cron, source-specific automation, and refresh automation later.

## 2. What PR-101 is now

PR-101 preview samples are manual preview only. They are not production timetable coverage.

The PR-101 preview page remains useful for checking display shape, source-link behavior, and static manual sample presentation. It must not be treated as the actual 13-country timetable product, a complete country/date/racecourse dataset, live coverage, or an automated refresh system.

## 3. 5PR schedule

### PR-102

- Build timetable ingestion model and source plan.
- Fix scope: 13 countries, 24 active groups, Singapore legacy.

### PR-103

- Add annual/season fixture records for all official-source-available records in the v0 coverage window.
- Do not use one sample per group.
- Unresolved or unstructured items remain visible with explicit status.

### PR-104

- Add rolling/racecard/entries/declarations/daily info records.
- Add first-race time and all-race times where official data is available.
- If racecard is not yet published, preserve the annual fixture with pending status.

### PR-105

- Merge annual fixtures and rolling/racecard data into `data/static/major-country-timetable-v0.json`.
- Add `/major-countries/timetable/` UI.
- Show country/system/date/racecourse/first-race/all-race-times/source links/source type/capture date/last checked/stale or pending status.

### PR-106

- Add completion gate for static/manual-driven timetable v0.
- Completion must be based on actual timetable behavior, not preview completion.

## 4. User-view outcome after PR-106

After PR-106, users should be able to:

- choose a country;
- choose a racing system/operator;
- choose a date;
- see racecourses with scheduled meetings;
- see first-race time where official racecard/entries/daily info exists;
- see all race times where official racecard/entries/daily info exists;
- see pending status where racecard is not yet published;
- see official source links;
- see whether the record came from annual fixture data or rolling/racecard data;
- see source capture date, last checked date, and stale status.

## 5. Operations outcome after PR-106

After PR-106, maintainers should have:

- a static/manual-driven 13-country timetable v0;
- separate annual fixture data;
- separate rolling/racecard data;
- merged display timetable data;
- source trace for each displayed item;
- visible pending/stale/manual-review states;
- a clear next path toward parser implementation, cron refresh, stale automation, and source-specific update logic.

## Acceptance criteria

- `data/static/major-country-timetable-ingestion-v0.json` exists.
- It covers all 13 countries.
- It covers all 24 active acquisition groups plus Singapore legacy.
- Every record has the required fields.
- Source policy and v0 coverage policy are explicit.
- PR-103 and PR-104 cannot be interpreted as one-sample-per-group tasks.
- Validation script exists and is wired into `npm run check`.
- This runbook explains the 5PR path, user-view outcome, and operations outcome.
- Preview/sample wording is clearly demoted from production timetable meaning.
- `npm run check` passes.

## Source and coverage policy notes

Annual fixtures are candidate meeting sources until rolling updates or racecard/entries/declarations/daily info confirm final first-race and all-race times. If racecard-level data is not yet officially published, annual fixture records remain visible with explicit statuses such as `pending_racecard`, `time_pending_official_racecard`, `source_not_yet_published`, or `needs_manual_review`.

Unresolved items must not be dropped. Missing structured source data must be represented explicitly with status, not silently excluded.
