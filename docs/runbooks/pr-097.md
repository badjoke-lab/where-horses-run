# PR-097: Manual source snapshot schema and first batch plan

## Summary

PR-097 starts the manual static source snapshot phase after PR-096 by defining the record shape and first batch plan that later PRs will use for official or approved source confirmation.

Users still cannot see real major-country timetable data from PR-097 alone. PR-097 prepares the schema and first batch plan only; it does not capture source evidence, generate timetable records, or complete a public timetable UI.

## User-visible meaning

PR-097 does not make real major-country timetable data visible to users.

The user-visible meaning is that the project now has the exact static structure needed to begin recording official-source confirmations. After the next capture PR, users should be closer to seeing real sample meeting dates, first-race times, per-race times, and official source links for the selected first-batch groups.

## Operational meaning

Operationally, PR-097 confirms that maintainers can begin manual static source capture without adding runtime extraction machinery. The new files define what evidence must be recorded, which source types are allowed, how normalized samples should be shaped, and which selected groups should be captured first.

Annual fixture pages are not final timing truth. Annual or season fixture pages may establish coarse schedule context, but rolling racecard, entries, declarations, or daily race information are required for final first-race and per-race timing confirmation.

## Manual source snapshot schema

The schema in `data/static/manual-source-snapshot-schema.json` defines the required record fields for future manual source snapshots, including:

- Snapshot identity and source metadata.
- Country and acquisition group identifiers.
- Source type and capture date.
- Source-local date context.
- Racecourse or meeting context.
- Annual or rolling context.
- First-race and per-race timing evidence.
- A normalized sample shape.
- Source trace metadata.
- User-visible result text.
- Reviewer notes.
- Snapshot status.

The schema allows source types such as annual fixtures, season fixtures, rolling fixtures, racecards, entries, declarations, daily race information, operator calendars, racecourse inventory, and legacy inventory.

## First batch plan

The first batch plan in `data/static/manual-source-snapshot-first-batch-plan.json` is `manual-source-snapshot-batch-001` and has `planned` status.

Its goal is to capture the first verified manual static source snapshots for selected high-confidence groups so real official-source confirmation can begin in PR-098.

The batch validates that official source evidence can be recorded without adding live fetching, parser runtime, scraping dependencies, raw source storage, generated live timetable records, or betting-adjacent data.

## Selected groups and reason

The selected first-batch groups are:

- Japan / JRA.
- Japan / NAR.
- Japan / Banei.
- Hong Kong / HKJC.
- United Arab Emirates / ERA.
- South Korea / KRA.

These groups were chosen because they are mostly centralized operator or authority sources. They are better suited for validating the manual snapshot workflow before moving into heavily operator-split countries such as North America. They also cover annual fixture, rolling fixture, daily race info, racecard, and season/declaration-style source patterns.

Each selected group requires evidence for:

- Racecourse or meeting context.
- Annual or rolling fixture context.
- First-race time.
- Per-race time.
- Official source URL.
- Source capture date.

## Singapore legacy handling

Singapore is not included inside `selected_groups`.

Singapore is represented only by a separate `singapore_status` object with `legacy_no_active_racing`, `manual_snapshot_required: false`, and a next action to keep legacy inventory only unless racing resumes or an approved successor source exists.

Singapore is not an active timetable target for this batch.

## What this PR does not do

No odds, payouts, predictions, or tips are in scope for this phase.
