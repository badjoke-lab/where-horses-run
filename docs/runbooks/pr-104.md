# PR-104: Major-country rolling racecards v0

PR-104 adds the rolling/racecard confirmation layer for the same major-country timetable v0 window used by PR-103:

- Start date: `2026-05-30`
- End date: `2026-06-30`
- Basis: `static_manual_v0_window`

The new dataset is `data/static/major-country-rolling-racecards-v0.json`.

## 1. What PR-104 adds

PR-104 adds a static/manual rolling layer that sits after the PR-102 source plan and the PR-103 annual fixture layer.

It includes:

- Rolling/racecard/entries/declarations/daily-info records for the v0 window when official source evidence is available.
- First-race time and per-race time rows where official sources make those times available for record-level capture.
- Pending records for PR-103 annual candidate meetings where official racecard data is not yet published.
- Coverage gaps where official rolling sources are unavailable, not structured for this PR, or require manual review before promotion.
- Source trace fields tying confirmed rolling records back to both:
  - `major-country-timetable-ingestion-v0`
  - `major-country-annual-fixtures-v0`

## 2. What PR-104 does not add

PR-104 intentionally does not add:

- No live fetch.
- No parser.
- No cron.
- No odds.
- No payouts.
- No predictions.
- No tips.
- No full racecard republication.
- No claim of full automation or live coverage.

The dataset is a static/manual v0 handoff layer. It does not replace source-specific parsing or scheduled refresh work.

## 3. User-view outcome after PR-104

After PR-104, users and maintainers can know:

- Which candidate meetings from PR-103 now have official time confirmation.
- First-race time where official racecard, entries, declarations, fields, acceptances, or daily information exists.
- Per-race times where official racecard, entries, declarations, fields, acceptances, or daily information exists.
- Which meetings remain pending because the racecard is not yet published.
- Which official source confirmed or failed to confirm the meeting time.

The unresolved annual fixture candidates remain visible as `pending_records[]` rather than being silently dropped.

## 4. Operations outcome after PR-104

After PR-104, maintainers have:

- Annual fixture candidate data from PR-103.
- Rolling/racecard confirmation data from PR-104.
- Pending states for unresolved meetings.
- Explicit source trace for final times.
- Input data ready for PR-105 merging into the user-facing timetable.

Operationally, the important behavior is preservation:

- Confirmed records go in `records[]`.
- PR-103 annual candidate meetings without official rolling timing evidence go in `pending_records[]`.
- Groups whose official rolling source cannot provide promoted records in this PR remain in `coverage_gaps[]`.
- Singapore remains `legacy_no_active_racing` and is not counted as an active acquisition group.

## 5. Validation

The validator is `scripts/check-major-country-rolling-racecards-v0.mjs` and is wired into:

```bash
npm run validate:major-country-rolling-racecards-v0
npm run check
```

The validator checks that:

- The rolling dataset exists.
- The coverage window matches PR-103.
- All 13 countries are represented.
- All 24 active acquisition groups are represented across confirmed records, pending records, or coverage gaps.
- Singapore legacy exists as no active racing and is not counted as active.
- Confirmed records have first-race time, race rows, source metadata, source trace, and annual matching metadata.
- Pending records use `final_times_status: pending_racecard`.
- Annual candidate meetings from PR-103 are preserved as pending records unless promoted separately.
- The dataset does not use fake/generated time status, public live-coverage claims, full-automation claims, or preview/sample completion wording.
