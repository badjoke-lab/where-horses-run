# PR-107 refresh operations plan

## Core decision

The project cannot be operated by manually editing meeting records one by one.

Manual work is allowed only for initial source validation, parser fixes, and review of generated diffs. Normal updates must be generated by scripts from a source registry.

## Target operating model

The update flow should be:

1. Source registry
2. Fetch official source snapshots
3. Parse source-specific rows
4. Normalize rows into common timetable records
5. Promote records by data depth
6. Build aggregate timetable data
7. Validate output
8. Create a data update PR
9. Human review only exceptions and suspicious diffs

## Data promotion model

- D: official source known, extraction not yet working
- C: official meeting date and racecourse captured
- B: first race time captured
- A: all race times captured

Annual or season fixture sources create Level C records.
Rolling fixture, entries, declarations, racecard, or daily race info sources confirm, correct, or promote those records to B or A.

## Annual fixture handling

Annual and season schedules are not final timetables. They are the base calendar.

They should be fetched in batches by country/system and stored as C-level records with source trace.

When a later rolling/racecard source exists for the same country, system, racecourse, and meeting date, the later source must override or confirm the annual fixture.

## Automated refresh jobs

### Job 1: annual fixture refresh

Purpose: maintain the base calendar.

Default cadence: monthly, plus manual run when a new season calendar is published.

Output:

- new C records
- changed C records
- removed/cancelled meeting candidates
- source capture report

### Job 2: next 30 days rolling refresh

Purpose: keep near-term meetings useful.

Default cadence: weekly.

Scope: meetings in the next 30 days.

Output:

- C confirmations
- B promotions when first race time appears
- changed meeting date/racecourse warnings

### Job 3: next 7 days racecard refresh

Purpose: capture race times before the meeting.

Default cadence: daily.

Scope: meetings in the next 7 days.

Output:

- B records when first race time is available
- A records when all race rows are available
- source trace for the final time source

### Job 4: day-of-race refresh

Purpose: catch late changes.

Default cadence: daily during active windows, source-friendly rate limited.

Output:

- updated A records
- changed race time warnings
- cancellation or abandonment markers when official source shows them

## Human work that remains

Human review should be limited to:

- adding a new official source to the registry
- fixing a parser when a site layout changes
- reviewing generated PR diffs
- checking suspicious changes, such as many cancelled meetings or large race-time shifts
- deciding whether a source should be trusted when two official pages conflict

Human work should not be:

- manually typing every meeting
- manually typing every race time
- manually refreshing all countries one by one

## Repository implementation path

### Step 1: source registry

Create one registry file containing every country/system source, source type, fetch cadence, parser name, and promotion target.

### Step 2: fetch snapshots

Create scripts that download official pages into timestamped snapshot files or cache artifacts.

### Step 3: parsers

Create one parser per source family. Each parser returns raw normalized rows.

### Step 4: normalizer/promoter

Match annual C records with rolling/racecard rows and output the best available record.

### Step 5: generated PR

Generated files should be committed in a data update PR with a summary:

- new records
- promoted records
- stale records
- cancelled/changed records
- parser errors

## Practical initial priority

PR-107 should only define this operating model and clean staging data.

Next PRs should add the pipeline in this order:

1. source registry
2. Japan NAR/Banei fetch + parser because A-level data is already proven
3. HKJC fixture/racecard promotion
4. BHA/HRI annual fixture parser
5. Canada/Australia/New Zealand source promotion from D to C
6. remaining D systems

## User-facing display rule

Users should not see internal parser noise.

They should see:

- data level
- meeting date
- racecourse
- first race time when available
- all race times when available
- annual source when the record is still C
- final time source when B/A exists
- last checked
- stale/needs refresh label when applicable

## Non-goals

This plan does not add odds, payouts, predictions, tips, betting advice, live coverage, or a full automation claim.
