# PR-109 — Timetable refresh and promotion command skeleton

## Purpose

PR-109 moves the project away from manually adding timetable JSON records as the normal operating model.

It introduces the source registry and command skeleton needed to keep rank promotion and race-time updates running through the same pipeline.

## Core model

Normal operation should be:

1. Source registry
2. Fetch or snapshot official pages
3. Parse
4. Normalize
5. Promote D/C/B/A
6. Build current timetable
7. Validate
8. Generate a data update PR

PR-109 does not implement live fetching or source-specific parsers. It fixes the command contracts and generated output contracts first.

## Commands added

- `refresh:timetable:annual`
- `refresh:timetable:rolling`
- `refresh:timetable:race-times`
- `promote:timetable`
- `archive:timetable`
- `build:timetable`
- `report:timetable-update`
- `refresh:timetable`

## Command meaning

### annual

Annual and season sources create or refresh Level C records.

Main promotion path:

- D -> C

### rolling

Rolling, fields, entries, and declarations sources keep near-term meetings fresh.

Main promotion path:

- C -> B
- C -> A when enough rows exist

### race-times

Race time / racecard / daily info sources establish final timing.

Main promotion path:

- B -> A
- C -> A

### promote

Promotion is the command that keeps retrying B-or-lower records.

It should include:

- D
- C
- B
- A upcoming
- A race_day

It should exclude:

- A archived
- cancelled archived

### archive

Archive fixes records after final confirmation.

A records are not permanently skipped until the meeting is complete and final checked.

## Source registry

Added:

- `data/source-registry/major-country-sources.json`

The registry keeps the 13-country / 24 active group / 1 legacy group scope and stores source kind, URL, parser name, target level, cadence, and status.

## Generated outputs

The skeleton commands write these contracts:

- `data/generated/timetable/current.json`
- `data/generated/timetable/update-report.json`
- `data/generated/timetable/source-health.json`
- `data/generated/timetable/promotion-status.json`

For PR-109 these are skeleton outputs only and must remain `skeleton_no_live_fetch`.

## Validation

Added:

- `scripts/check-timetable-refresh-skeleton.mjs`
- `validate:timetable-refresh-skeleton`

The validator checks:

- registry schema
- 13 countries
- 24 active groups
- 1 legacy group
- valid source kinds
- valid target levels
- command files exist
- skeleton commands execute
- generated output contracts exist

## Non-goals

PR-109 does not:

- complete any country parser
- perform live source fetches
- create automated update PRs
- finish D-to-C coverage
- claim full automation
- add odds, payouts, predictions, tips, or betting advice
