# PR-078: Cross-country candidate validator

## Summary

PR-078 adds a cross-country candidate validator for the shared timetable candidate invariants used by Japan, Hong Kong, and UAE candidate and approved-bundle files before the M3 v0 release gate.

The validator is intentionally local-only. It reads JSON candidate files that already exist in the repository, validates their shared contract, applies a small country-specific layer, and exits non-zero on unsafe or inconsistent candidate content.

## Files

- `scripts/check-cross-country-candidate-validator.mjs` — shared validator for cross-country candidate and approved-bundle invariants.
- `package.json` — adds `validate:cross-country-candidate-validator` and includes it in `npm run check`.
- `docs/runbooks/pr-078.md` — this runbook.

## User-facing outcome

No direct UI change.

This PR adds a release-safety validation step only. It does not replace public overlays, promote data, or change rendered timetable pages.

## Safety boundary

The validator must remain a structural and safety-boundary check only:

- No live fetch.
- No source page parsing.
- No raw source body storage.
- No racecard, odds, results, payouts, predictions, or tips.
- No public overlay replacement.
- No UI changes.
- No new dependencies.

Candidate files fail validation if they contain forbidden markers such as raw HTML, source bodies, racecards, entries, horse names, jockey names, trainer names, odds, results, payouts, predictions, or tips.

## Covered candidate files

The validator checks these files when present:

- `data/candidates/japan-active-window-approved-candidates.json`
- `data/candidates/hong-kong-active-window-approved-candidates.json`
- `data/candidates/uae-active-window-approved-candidates.json`
- `data/candidates/hong-kong-hkjc-candidates.json`
- `data/candidates/uae-era-candidates.json`

Shared validation covers:

- `schema_version: timetable-candidates-v0`
- top-level `country_id`, `source_adapter_id`, `candidate_window`, `records`, and `review`
- allowed top-level review status values: `needs_review` and `approved`
- record identity and provenance fields
- status/review-status pairing
- duplicate `candidate_id` detection
- duplicate logical-key detection across `country_id`, `date`, `racing_system_id`, `racecourse_id`, and `source_id`

Country-specific validation covers:

- Japan approved bundles remain `country_id: japan`, source-reviewed, approved, and include JRA, NAR, and Banei when records are present.
- Hong Kong files remain `country_id: hong-kong` with `Asia/Hong_Kong`; approved bundles include Happy Valley and Sha Tin when records are present.
- UAE files remain `country_id: united-arab-emirates` with `Asia/Dubai`; approved bundles may preserve an empty `records: []` season gap.

## Season gap handling

The UAE approved bundle is allowed to keep `records: []` for the current season gap.

When the UAE approved bundle is empty, the validator requires visible season-gap wording and explicit `no active-window` wording. It also requires the file to avoid public coverage claims and to keep `review.promotion_target` as `null`.

This preserves the PR-077 behavior: an empty UAE active-window approved candidate bundle can document a season gap without becoming public timetable coverage.

## Validation commands

```bash
node scripts/check-cross-country-candidate-validator.mjs
npm run build
npm run check
```
