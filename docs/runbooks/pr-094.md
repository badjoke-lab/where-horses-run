# PR-094 — Major-country sample extraction snapshots

## Summary

PR-094 adds static sample extraction snapshots for the major-country acquisition scope. The snapshots document the expected extracted fields for each PR-091 sample fixture while staying within the PR-093 parser contracts and normalization rules.

These snapshots are examples of contract-level extraction shape only. They do not fetch live pages, parse live source documents, store raw source bodies, or create generated timetable records.

## Files

- `data/static/major-country-sample-extraction-snapshots.json` — static extraction snapshot inventory keyed to the PR-091 sample fixtures.
- `scripts/check-major-country-sample-extraction-snapshots.mjs` — validator for fixture coverage, parser-contract references, placeholder-only extraction fields, Singapore legacy handling, and prohibited live/runtime additions.
- `docs/runbooks/pr-094.md` — this runbook.
- `package.json` — adds the validation script and wires it into `npm run check` immediately after `validate:major-country-parser-contracts`.

## Relationship to PR-091 sample fixtures

The extraction snapshot file references `data/static/major-country-sample-acquisition-fixtures.json` and must mirror its fixture coverage exactly:

- all 13 major-country scope entries are present;
- every PR-091 `sample_fixtures[]` entry has exactly one PR-094 `extraction_snapshots[]` entry;
- each snapshot repeats the fixture identity fields:
  - `country_id`
  - `group_id`
  - `parser_target`
  - `sample_target_type`
  - `sample_source_url`
- `source_trace.source_url`, `source_trace.source_role`, and `source_trace.source_group_id` point back to the same PR-091 fixture source, target role, and group.

## Relationship to PR-093 parser contracts

The extraction snapshot file references `data/static/major-country-parser-contracts.json`. Each non-legacy extraction snapshot must map to a PR-093 parser contract by:

- `country_id`
- `group_id`
- `parser_target`

PR-094 does not add parser logic. It only records a static expected extraction example for the existing parser-contract target.

## Static extraction snapshot shape

Each non-legacy extraction snapshot uses this shape:

```json
{
  "country_id": "<country_id>",
  "group_id": "<group_id>",
  "parser_target": "<parser_target>",
  "sample_target_type": "<sample_target_type>",
  "sample_source_url": "<sample_source_url>",
  "extraction_snapshot_kind": "static_contract_extraction_example",
  "extraction_status": "static_expected_extraction_only",
  "extracted_fields": {
    "racecourse": "<racecourse>",
    "meeting_date": "<meeting_date>",
    "first_race_time": "<first_race_time>",
    "races": [
      {
        "race_number": "<race_number>",
        "race_time": "<race_time>",
        "race_name_or_label": "<race_name_or_label>"
      }
    ]
  },
  "source_trace": {
    "source_url": "<sample_source_url>",
    "source_role": "<sample_target_type>",
    "source_group_id": "<group_id>"
  },
  "contract_reference": "data/static/major-country-parser-contracts.json",
  "notes": "Static placeholder extraction snapshot for the PR-091 sample fixture under the PR-093 parser contract."
}
```

The `races[]` array intentionally contains exactly one placeholder object. It is a schema-shape example, not a generated list of live race rows.

## Singapore legacy handling

Singapore remains legacy inventory only:

- the file includes exactly one Singapore extraction snapshot;
- its `extraction_status` is `legacy_inventory_only`;
- its `extracted_fields` include `racecourse` and `meeting_date` placeholders only;
- it does not include `first_race_time`;
- it does not include `races[]`;
- it does not create active timetable output.

## What this PR does

- Adds placeholder-only extraction snapshots for the major-country PR-091 fixture set.
- Links non-legacy snapshots to the PR-093 parser-contract file.
- Adds validation that the extraction snapshot inventory remains static and fixture-complete.
- Adds validation that Singapore remains legacy inventory only.

## What this PR does not do

- No live fetch runtime.
- No parser implementation.
- No scraping dependency.
- No raw source body or HTML storage.
- No generated live timetable records.
- No odds.
- No payouts.
- No predictions.
- No tips.
- No new dependencies.
