# Data model specification

Status: draft  
Scope: v0 static MVP and GitHub Actions generated data

This document defines the initial data model for Where Horses Run / 競馬どこ？.

The model is split into two groups:

```text
data/static/      human-maintained source data
data/generated/   GitHub Actions generated runtime data
```

The project is static-first. Data should be easy to validate, easy to review in pull requests, and safe to render without a server database.

---

## 1. Core rules

### 1.1 Stable IDs

Every primary object must have a stable lowercase ID.

```text
japan
hong-kong
tokyo-racecourse
sha-tin
harness-racing
```

IDs should not change after public release unless absolutely necessary.

### 1.2 Slugs

Slugs are URL-facing strings.

```text
/countries/japan/
/tracks/sha-tin/
/glossary/harness-racing/
```

Usually `id` and `slug` are the same. Keep both fields to allow future redirects.

### 1.3 Locale fields

English is required. Japanese is optional in data, but required for key UI pages in v0.

```ts
name_en: string
name_ja?: string
name_local?: string
```

### 1.4 Timezone fields

Use IANA timezone IDs only.

```text
Asia/Tokyo
Asia/Hong_Kong
America/Santiago
Europe/London
Australia/Sydney
Africa/Casablanca
```

Do not store fixed offsets like `UTC+9` as the source of truth.

### 1.5 Official links

Official links should point to the most stable official or authority source available.

If race-level data is not safe or available, keep only the official source link.

```text
data/
  static/

      en.json

    latest.json
```

### 2.1 `data/static/`

Human-maintained data. Changes should happen through normal pull requests.

### 2.2 `data/generated/`

Generated by GitHub Actions. These files may be updated automatically, but only when generated content changed.

## 3. Common enums

### 3.1 Country status

```ts
type CountryStatus =
  | "active"
  | "under_review"
  | "archive"
  | "excluded"
  | "special";
```

### 3.2 Racecourse status

```ts
type RacecourseStatus = "active" | "archive" | "unknown";
```

### 3.3 Coverage level

```ts
type CoverageLevel = 1 | 2 | 3 | 4 | 5;
```

Meaning:

| Level | Meaning |
|---|---|
| 1 | country / track / official links only |
| 2 | fixture or calendar dates |
| 3 | race-day timetable |
| 4 | race name, distance, surface, or minimal detail |
| 5 | reserved, not used in v0 without permission |

### 3.4 Auto level

```ts
type AutoLevel = "A" | "B" | "C" | "D" | "X";
```

Meaning:

| Level | Meaning |
|---|---|
| A | stable source, good for scheduled updates |
| B | update possible, parser and fallback needed |
| C | semi-manual or link-first source |
| D | not suitable for automation yet |
| X | archive / closed / no active update |

### 3.5 Source type

```ts
type SourceType =
  | "official"
  | "semi_official"
  | "authority"
  | "news"
  | "social"
  | "archive";
```

### 3.6 Source data type

```ts
type SourceDataType =
  | "calendar"
  | "racecard"
  | "programme"
  | "results"
  | "link_only";
```

`results` can exist as a source type for classification, but v0 must not republish detailed result data.

### 3.7 Terms risk

```ts
type TermsRisk = "low" | "medium" | "high" | "unknown";
```

Use `unknown` until reviewed.

## 4. Country

Stored in:

```text
data/static/countries.json
```

Shape:

```ts
type Country = {
  id: string;
  slug: string;
  name_en: string;
  name_ja?: string;
  name_local?: string;
  region: string;
  status: CountryStatus;
  racing_types: string[];
  coverage_level: CoverageLevel;
  auto_level: AutoLevel;
  available_locales: string[];
  summary_en?: string;
  summary_ja?: string;
  notes?: string;
};
```

Example:

```json
{
  "id": "hong-kong",
  "slug": "hong-kong",
  "name_en": "Hong Kong",
  "name_ja": "香港",
  "name_local": "香港",
  "region": "Asia",
  "status": "active",
  "racing_types": ["thoroughbred-flat"],
  "coverage_level": 3,
  "auto_level": "B",
  "available_locales": ["en", "ja"],
  "summary_en": "Major racing jurisdiction with official racecourse and race-day sources."
}
```

Validation:

- `id`, `slug`, `name_en`, `region`, `status`, `coverage_level`, `auto_level` are required.
- `coverage_level` must be 1-5.
- `auto_level` must be A/B/C/D/X.
- `available_locales` must include `en`.

## 5. Racecourse

Stored in:

```text
data/static/racecourses.json
```

Shape:

```ts
type Racecourse = {
  id: string;
  slug: string;
  country_id: string;
  name_en: string;
  name_ja?: string;
  name_local?: string;
  city?: string;
  official_url?: string;
  timezone: string;
  racing_types: string[];
  status: RacecourseStatus;
  image?: TrackImage;
  summary_en?: string;
  summary_ja?: string;
};
```

Example:

```json
{
  "id": "sha-tin",
  "slug": "sha-tin",
  "country_id": "hong-kong",
  "name_en": "Sha Tin",
  "name_ja": "沙田競馬場",
  "name_local": "沙田馬場",
  "city": "Hong Kong",
  "official_url": "https://racing.hkjc.com/",
  "timezone": "Asia/Hong_Kong",
  "racing_types": ["thoroughbred-flat"],
  "status": "active"
}
```

Validation:

- `country_id` must match an existing country.
- `timezone` must be an IANA timezone.
- `official_url`, if present, must be a valid URL.

## 6. Source

Stored in:

```text
data/static/sources.json
```

Shape:

```ts
type Source = {
  id: string;
  country_id: string;
  racecourse_id?: string;
  source_type: SourceType;
  url: string;
  data_type: SourceDataType;
  auto_level: AutoLevel;
  terms_risk: TermsRisk;
  update_frequency?: string;
  fallback_url?: string;
  notes?: string;
};
```

Example:

```json
{
  "id": "hong-kong-hkjc-racecard",
  "country_id": "hong-kong",
  "racecourse_id": "sha-tin",
  "source_type": "official",
  "url": "https://racing.hkjc.com/",
  "data_type": "racecard",
  "auto_level": "B",
  "terms_risk": "unknown",
  "update_frequency": "daily on race days"
}
```

Validation:

- `country_id` must exist.
- `racecourse_id`, if present, must exist.
- `url` must be valid.
- `terms_risk` defaults to `unknown` until reviewed.

## 7. Meeting

Generated file candidates:

```text
data/generated/today.json
data/generated/tomorrow.json
data/generated/calendar-30d.json
```

Shape:

```ts
type Meeting = {
  id: string;
  country_id: string;
  racecourse_id: string;
  date_local: string;
  timezone: string;
  source_id: string;
  status: "scheduled" | "cancelled" | "unknown";
  last_checked_at: string;
  races?: Race[];
};
```

Rules:

- `date_local` is the racecourse local date.
- `timezone` is copied from the racecourse/source context.
- `last_checked_at` should be stored as ISO datetime.

---

## 8. Race

Nested under `Meeting.races` in generated data.

Shape:

```ts
type Race = {
  id: string;
  meeting_id: string;
  race_number: string;
  start_date_local: string;
  start_time_local?: string;
  start_timezone: string;
  start_at_utc?: string;
  race_name?: string;
  distance?: string;
  surface?: string;
  official_detail_url?: string;
  coverage_level: CoverageLevel;
  status: "scheduled" | "delayed" | "cancelled" | "unknown";
};
```

Rules:

- Official source time is treated as local track time.
- `start_at_utc` is generated only when local date, local time, and timezone are available.
- Do not store JST as source-of-truth.
- If time is unknown, do not guess.

Example:

```json
{
  "id": "sha-tin-2026-05-27-r1",
  "meeting_id": "sha-tin-2026-05-27",
  "race_number": "1R",
  "start_date_local": "2026-05-27",
  "start_time_local": "20:10",
  "start_timezone": "Asia/Hong_Kong",
  "start_at_utc": "2026-05-27T12:10:00Z",
  "official_detail_url": "https://racing.hkjc.com/",
  "coverage_level": 3,
  "status": "scheduled"
}
```

## 9. FetchStatus

Stored in:

```text
data/generated/fetch-status.json
```

Shape:

```ts
type FetchStatus = {
  source_id: string;
  status: "ok" | "failed" | "stale" | "manual" | "skipped";
  checked_at: string;
  message?: string;
  fallback_url?: string;
};
```

Rules:

- Failed fetches must not break static rendering.
- UI must show fallback source links where available.
- Unknown data must not be filled by guessing.

## 11. GlossaryImage

Shape:

```ts
type GlossaryImage = {
  src: string;
  alt_en: string;
  alt_ja?: string;
  image_type: "generated_png" | "placeholder";
  status: "planned" | "generated" | "approved";
};
```

Rules:

- PNG only for final explanatory images.
- SVG is not used for glossary explanation assets.
- `alt_en` is required when `src` is present.
- Initial data may use planned image metadata without a real image file.

## 12. TrackImage

Shape:

```ts
type TrackImage = {
  src: string;
  alt_en: string;
  alt_ja?: string;
  image_type: "illustrative_generated_png" | "placeholder";
  is_official_photo: false;
  note_en: "Illustrative image. Not an official venue photo.";
  note_ja: "説明用のイメージ画像です。公式写真ではありません。";
  status: "planned" | "generated" | "approved";
};
```

Rules:

- Track images are illustrative, not official photos.
- Do not reproduce venue photos, logos, sponsor boards, real people, or real horses.
- Image captions must make the illustrative status clear.

## 14. i18n dictionary

Stored in:

```text
data/static/i18n/en.json
data/static/i18n/ja.json
```

Rules:

- UI strings should not be hardcoded in page templates.
- English root is required.
- Japanese is required for the initial public navigation and core notices.
- Future locales can be added without changing data shapes.

Example:

```json
{
  "site.title": "Where Horses Run",
  "nav.countries": "Countries",
  "nav.tracks": "Tracks",
  "nav.glossary": "Glossary",
  "label.officialSource": "Official source"
}
```

## 16. Validation rules

Initial validator should check:

- required fields are present
- IDs are unique within each file
- slugs are unique within each file
- country references exist
- racecourse references exist
- source references exist
- URLs are valid
- coverage levels are valid
- auto levels are valid
- IANA timezone strings exist
- glossary related terms exist
- image fields have alt text when `src` is present

---

## 17. Data change policy

### 17.1 Human-maintained static data

Changes must be made through normal pull requests.

### 17.2 Generated data

GitHub Actions may update `data/generated/`.

Rules:

- commit only when generated files changed
- do not overwrite `data/static/` from Actions
- fetch failures should produce `fetch-status.json`, not broken builds

---

## 18. v0 implementation priority

Implement in this order:

```text
1. static schema files
2. static data skeleton
3. validation scripts
4. Astro data loader
5. list/detail pages
6. generated data loader
7. GitHub Actions update flow
```

The first implementation does not need live fetching. It only needs to validate and render the static data skeleton.
